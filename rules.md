1. Busca siempre la solución más simple y directa posible (principio KISS)
2. Prefiere código legible sobre código "ingenioso" o excesivamente compacto
3. Evita la sobreingeniería - no agregues características no solicitadas
4. Implementa solo lo necesario para resolver el problema actual
5. Usa nombres de variables descriptivos y claros
6. Mantén las funciones pequeñas con responsabilidades únicas
7. Comenta solo cuando el código no sea autoexplicativo
8. Sigue las convenciones del lenguaje/framework que estés usando
9. Mantén consistencia con el código existente del proyecto
10. Usa indentación y formato consistentes
11. Implementa manejo de errores en todas las operaciones críticas
12. Proporciona mensajes de error claros y útiles
13. Valida las entradas antes de procesarlas
14. Usa try-catch/límites de error según sea apropiado
15. No ocultes errores silenciosamente
16. Optimiza solo cuando sea necesario y medible
17. No sacrifiques legibilidad por micro-optimizaciones prematuras
18. Considera el impacto en los recursos (memoria, CPU)
19. Evita operaciones costosas innecesarias en bucles
20. Usa estructuras de datos apropiadas para cada caso
21. Nunca expongas credenciales o información sensible en el código
22. Valida y sanea todas las entradas del usuario
23. Usa prácticas seguras (previene inyección SQL, XSS, etc.)
24. No confíes en datos del cliente sin validación
25. Implementa autenticación y autorización cuando sea necesario
26. Confirma que entiendes el problema correctamente
27. Propón la solución más simple que resuelva el requerimiento
28. Si hay múltiples enfoques, explica opciones y compensaciones
29. Pregunta si algo no está claro antes de comenzar
30. Escribe código incremental y testeable
31. Refactoriza cuando veas código duplicado o confuso
32. Prueba el código antes de considerarlo terminado
33. Explica lo que vas a hacer antes de hacerlo
34. Reporta problemas o limitaciones encontradas
35. Sugiere mejoras cuando sea apropiado
36. Sé claro sobre qué funciona y qué no
37. Verifica casos límite y situaciones de frontera
38. Prueba con datos reales cuando sea posible
39. Valida el comportamiento esperado antes de entregar
40. Documenta decisiones de diseño importantes
41. Mantén el README actualizado con instrucciones claras
42. Documenta APIs y funciones públicas
43. Explica el "por qué" no solo el "qué"
44. Crea los mensajes de commits en español
